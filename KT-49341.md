# KT-49341

## Summary
Passing `Class` instead of `KClass` to an overloaded function with a lambda crashes the K1 compiler with `NoSuchElementException: List is empty`.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt49341.kt`

## Observations
The ticket describes a K1 compiler crash in `NewCommonSuperTypeCalculator` when calling an overloaded function (one overload accepts `KClass<U>`, another accepts a lambda) with a `Class<T>` argument (`.java`) combined with a trailing lambda. In K1, this triggers `NoSuchElementException: List is empty` due to an empty list being passed to `types.first()`.

In FIR (K2), the compiler handles this correctly: calling `tasks.registering(MyTask::class.java) {}` produces a `NONE_APPLICABLE` diagnostic (no matching overload), which is the expected behavior described in the ticket. There is no crash.

The test passes cleanly with both the FIR analysis tests and the LL analysis tests, confirming the bug is not present in FIR.
