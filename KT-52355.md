# KT-52355

## Summary
Not correct specified type parameter in the error OVERLOAD_RESOLUTION_AMBIGUITY_BECAUSE_OF_STUB_TYPES — the error message shows a technical descriptor string like `<T> defined in itemHolderBuilder[TypeParameterDescriptorImpl@6dc675b2]` instead of just `T`.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt52355.kt`

## Observations
The issue was reported against K1 (Kotlin 1.6.21) where `println(lastItem)` inside a `@BuilderInference` lambda triggered `OVERLOAD_RESOLUTION_AMBIGUITY_BECAUSE_OF_STUB_TYPES` with a poorly formatted error message containing the internal `TypeParameterDescriptorImpl` descriptor representation.

In K2/FIR, the reproducer code compiles successfully to BACKEND with no ambiguity error. The type inference for `@BuilderInference` works correctly in K2 — the type parameter `T` is properly resolved to `Int` when `addItem(s: Int)` is called, and `println(lastItem)` is not ambiguous. The only diagnostic present is a `DEPRECATION` warning on the `@BuilderInference` annotation itself (which requires `@OptIn(ExperimentalTypeInference::class)`).

The `OVERLOAD_RESOLUTION_AMBIGUITY_BECAUSE_OF_STUB_TYPES` diagnostic from the ticket is not raised by the K2 frontend, so the bug (incorrect error message format) is not reproduced in K2/FIR.
