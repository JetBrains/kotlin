# KT-40687

## Summary
False-positive TYPE_MISMATCH: KClass<Captured(*)> is not a subtype of KClass<Any> in HMPP (Hierarchical Multiplatform Project) with granular source sets metadata enabled.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt40687.kt`

## Observations
The reproducer from the ticket:
```kotlin
import kotlin.reflect.KClass

fun foo(locationClass: KClass<*>?) {
    val type = locationClass ?: Any::class // TYPE_MISMATCH in K1+HMPP
}
```

When run as a FIR analysis test, the code compiles successfully to BACKEND with no diagnostics — no TYPE_MISMATCH is reported. This confirms that the false-positive error described in the ticket (where `KClass<Captured(*)>` was incorrectly flagged as not a subtype of `KClass<Any>`) is not present in the FIR (K2) frontend.

The original issue was specific to the K1 frontend in IDE mode with `kotlin.mpp.enableGranularSourceSetsMetadata=true`. Per the ticket comments, it was also noted to be "an error only in IDE highlighting" even for the K1 case. FIR handles this type inference correctly, so the bug is not reproduced.
