# KT-42108

## Summary
HMPP: NOTHING_TO_OVERRIDE instead of CONFLICTING_OVERLOADS when override declarations have a different number of type parameters and an `actual typealias` is involved.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt42108.kt`

## Observations

The ticket describes two cases:
1. **Non-MPP (K1)**: `override fun <T, R> foo(e: E)` where base has `fun <T> foo(e: E)` → emits `CONFLICTING_OVERLOADS` (correct behavior)
2. **HMPP/MPP (K1 bug)**: Same override but with `expect class E` / `actual typealias E = String` → wrongly emits `NOTHING_TO_OVERRIDE` instead of `CONFLICTING_OVERLOADS`

The root cause (K1): `isOverrideableByWithoutExternalConditions` uses a default type checker without the type refiner, so with `actual typealias E = String`, the E types appear not to match between common and platform modules → wrong diagnostic.

**FIR (K2) behavior on the non-MPP case**: FIR emits **both** `NOTHING_TO_OVERRIDE` **and** `CONFLICTING_OVERLOADS` on the override declaration:
```
<!NOTHING_TO_OVERRIDE!>override<!> <!CONFLICTING_OVERLOADS!>fun <T, R> foo(e: E)<!>
```

The ticket's expected (correct) behavior for non-MPP is `CONFLICTING_OVERLOADS` only (without `NOTHING_TO_OVERRIDE`). Since FIR still emits `NOTHING_TO_OVERRIDE` even in the non-MPP case, the bug is reproduced in FIR.

Note: A multiplatform test (with `expect class E` / `actual typealias E = String`) was also attempted but caused conflicts between parallel test runners (FirLightTreeDiagnosticsWithoutAliasExpansionTestGenerated vs PhasedJvmDiagnosticLightTreeTestGenerated) due to different diagnostic expectations with/without alias expansion. The non-MPP test was used as the reproducer.
