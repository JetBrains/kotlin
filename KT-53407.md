# KT-53407

## Summary
`HashMap<…>` not recognized as subtype of Java's raw `Map` — false positive `JAVA_TYPE_MISMATCH` when passing `HashMap<String, String>` to a Java method expecting raw `Map`.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt53407.kt`

## Observations
The test creates a Java class with a constructor and static method accepting raw `Map`, then calls them from Kotlin with:
1. `HashMap<String, String>` — the ticket says this should give `JAVA_TYPE_MISMATCH` (bug)
2. `Map<String, String>` — the ticket says this works fine

In the current FIR/K2 implementation, **no diagnostics are generated** for either call. Both `HashMap<String, String>` and `Map<String, String>` are correctly accepted as subtypes of a raw Java `Map`. The test compiles cleanly to `BACKEND` with no errors or warnings.

The bug described in the ticket (originally reported for Kotlin 1.6.21/1.7.0, K1 frontend) is **not present** in the current K2/FIR implementation. FIR correctly handles `HashMap<String, String>` as a valid subtype of Java's raw `Map` type.

Note: The ticket comment mentions "Reproduced only in an Android project," which may indicate the issue was specific to how Android libraries (e.g., `JSONObject` with `@NonNull` annotations) are processed in K1. This specific scenario was not tested here.
