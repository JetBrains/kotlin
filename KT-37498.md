# KT-37498

## Summary
NI: Misleading diagnostic message for incorrectly specified lambda parameter — the compiler reports `EXPECTED_PARAMETER_TYPE_MISMATCH` (misleading) instead of a clearer `TYPE_MISMATCH` when a lambda parameter type is explicitly annotated but the return type doesn't match.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt37498.kt`

## Observations
The minimized reproducer from the ticket comments:
```kotlin
fun <R> foo(op: (R) -> R) {}
val test = foo { _: String -> }
```

In FIR/K2, the compiler emits `ARGUMENT_TYPE_MISMATCH` on the entire lambda expression `{ _: String -> }` rather than a more precise diagnostic pointing to the mismatch between the inferred return type (Unit) and the expected return type (String/R). This confirms the issue is still present: the diagnostic is placed on the whole lambda argument rather than pinpointing where the type mismatch actually occurs (the missing return value), and the message ("argument type mismatch") is not as informative as it could be.
