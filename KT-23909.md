# KT-23909

## Summary
Use more specific diagnostic for wrong calls with implicit invokes — FIR produces a wrong/less specific diagnostic instead of `TYPE_MISMATCH`.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt23909.kt`

## Observations
The reproducer code is `1.(fun String.() = 1)()` — calling an extension function literal (`fun String.() = 1`) with `1` (Int) as the receiver, where `String` is expected.

- **Old inference (K1)**: reports `TYPE_MISMATCH` (correct — `Int` is not `String`)
- **FIR at time of ticket filing**: reported `FUNCTION_EXPECTED` (wrong, less specific)
- **Current FIR**: reports `UNRESOLVED_REFERENCE_WRONG_RECEIVER` on `(fun String.() = 1)`

The diagnostic is still not `TYPE_MISMATCH` as expected. The bug is still present — FIR uses a different (wrong) diagnostic code. The test passes by asserting the current erroneous behavior (`UNRESOLVED_REFERENCE_WRONG_RECEIVER`).
