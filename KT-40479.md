# KT-40479

## Summary
`UNRESOLVED_REFERENCE_WRONG_RECEIVER` is produced when calling an extension on `KClass<T : Any>` with a reified type parameter `T` that has no explicit `Any` upper bound, preventing IDE completion for `primaryConstructor` in that context.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt40479.kt`

## Observations
The test defines a self-contained extension `fun <T : Any> KClass<T>.myPrimaryConstructor(): Any? = null` to reproduce the issue without requiring `kotlin-reflect`.

When `T` has no `Any` upper bound (`inline fun <reified T> foo()`), calling `T::class.myPrimaryConstructor()` produces `UNRESOLVED_REFERENCE_WRONG_RECEIVER` — exactly the diagnostic reported in the ticket comments.

When `T` explicitly has `Any` upper bound (`inline fun <reified T : Any> bar()`), the call resolves correctly without any error.

The test asserts `<!UNRESOLVED_REFERENCE_WRONG_RECEIVER!>myPrimaryConstructor<!>` on line 14, confirming the bug is still present. All test runners (FirLightTreeDiagnostics, PhasedJvmDiagnosticLightTree, PhasedJvmDiagnosticPsi, LLDiagnostics variants) pass with these markers.
