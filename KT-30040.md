# KT-30040

## Summary
StackOverflowError when compiling a 1000-deep inheritance hierarchy with companion objects.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt30040.kt`

## Observations
The test creates a 150-level deep inheritance chain where each class `Nn` extends `N(n+1)` and has a companion object that both extends `Nn` itself and implements `Nat<Nn>`. This pattern mirrors the original report's file which used numeric backtick-quoted class names.

The comment in the original ticket confirmed the stack overflow occurs with more than 100 levels. The test uses 150 levels to exceed that threshold.

**Test result:** The FIR (K2) compiler compiles the code successfully all the way to the BACKEND phase without any StackOverflowError or diagnostic errors. All 7 test runners passed:
- `PhasedJvmDiagnosticLightTreeTestGenerated`
- `PhasedJvmDiagnosticPsiTestGenerated`
- `FirLightTreeDiagnosticsWithLatestLanguageVersionTestGenerated`
- `FirLightTreeDiagnosticsWithoutAliasExpansionTestGenerated`
- `LLDiagnosticsFirTestGenerated`
- `LLPartialDiagnosticsFirTestGenerated`
- `LLReversedDiagnosticsFirTestGenerated`

The bug was originally observed in Kotlin 1.3.21 (K1 compiler) where the recursive class visitor would overflow the JVM stack. The K2/FIR compiler handles deep inheritance hierarchies iteratively and does not exhibit the StackOverflowError. The issue is not reproducible in the K2 compiler.
