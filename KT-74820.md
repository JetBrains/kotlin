# KT-74820

## Summary
Investigate using `mergeOtherSystem` at PostponedArgumentsAnalyzer#applyResultsOfAnalyzedLambdaToCandidateSystem to replace the controversial `UnstableSystemMergeMode` opt-in API with `replaceContentWith`.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt74820.kt`

## Observations
KT-74820 is an internal compiler investigation task (type: Task, not Bug) about refactoring the constraint system merging in `PostponedArgumentsAnalyzer`. It does not describe any user-visible misbehavior — it proposes making `additionalConstraintStorage` a clone of the shared constraint system so that `replaceContentWith` can be used instead of `mergeOtherSystem` (which requires `@OptIn(UnstableSystemMergeMode::class)`).

The test was created to exercise lambda type inference scenarios that go through `PostponedArgumentsAnalyzer#applyResultsOfAnalyzedLambdaToCandidateSystem`:
- Simple lambda with inferred return type (`runWith`)
- Lambda with two type parameters (`transform`)
- Multiple lambdas with shared type variables (`combine`)
- Nested lambdas with constraint propagation (`runWith` + `transform`)
- Standard library lambda chains (`map` + `filter`)

All test cases compile successfully to BACKEND without errors or warnings. The FIR analysis produces no diagnostics, confirming that the current implementation works correctly for these lambda inference scenarios. Since this is a code quality/refactoring investigation rather than a bug, `repro: false` — there is no buggy behavior to reproduce.

Note: During test execution with `--update-test-data`, a race condition occurred between test runners (`FirLightTreeDiagnosticsTestGenerated` wrote `GENERATED_FIR_TAGS` and `PhasedJvmDiagnosticLightTreeTestGenerated` overwrote them). The tags were added manually from the first runner's output, and the final test run passed cleanly.
