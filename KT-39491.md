# KT-39491

## Summary
"Recursive call in a lazy value under LockBasedStorageManager" when accessing constructors of @Serializable classes via a `SyntheticResolveExtension` compiler plugin.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt39491.kt`

## Observations
The original issue was K1-specific: when any compiler plugin's `SyntheticResolveExtension.generateSyntheticMethods` called `ClassDescriptor#getConstructors` on a class annotated with `@Serializable`, the K1 `LockBasedStorageManager` would throw:
```
java.lang.IllegalStateException: Recursive call in a lazy value under LockBasedStorageManager
```

This crash is caused by the serialization compiler plugin's own synthetic resolution holding a lock while another plugin tries to access the same lazy value (constructors).

The FIR analysis test creates a class annotated with a custom `@Serializable` annotation that has multiple constructors (primary and secondary), mirroring the code pattern from the ticket. The test compiles cleanly to `BACKEND` with no diagnostics — FIR handles constructor resolution of annotated classes without any recursive lock issues, since FIR does not use `LockBasedStorageManager`. The bug is not reproduced in FIR/K2.

The test ran successfully (`resultType="SUCCESS"`) across all test runners:
- `PhasedJvmDiagnosticLightTreeTestGenerated`
- `PhasedJvmDiagnosticPsiTestGenerated`
- `LLDiagnosticsFirTestGenerated` and related LL API tests
