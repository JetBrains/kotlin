# KT-25808

## Summary
False negative "Operator '==' cannot be applied to" for platform types: `(1 to B()) == B()` where `B` is a Java class does not produce a compile-time error, while the equivalent `(1 to A()) == A()` with a Kotlin class does.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt25808.kt`

## Observations
- The test uses a Java class `B` and a Kotlin class `A` to compare `(1 to X()) == X()` (Pair<Int, X> vs X).
- In K1 (original report): `(1 to A()) == A()` produces `EQUALITY_NOT_APPLICABLE`, but `(1 to B()) == B()` does not — false negative for the platform type.
- In K2/FIR: **neither** expression produces a diagnostic. The `EQUALITY_NOT_APPLICABLE` check for incompatible types via `==` appears to be missing entirely in FIR, not just for platform types.
- The test passes with `RUN_PIPELINE_TILL: BACKEND` and no diagnostic markers, confirming FIR compiles both comparisons without error — matching the (buggy) behavior described in the ticket for the Java platform type case.
- `GENERATED_FIR_TAGS` added: `classDeclaration, equalityExpression, functionDeclaration, integerLiteral, javaFunction, javaType`
- All 7 test variants pass (FirLightTree, PhasedJvmPsi, PhasedJvmLightTree, LL variants).
