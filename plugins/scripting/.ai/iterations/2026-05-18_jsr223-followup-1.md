## Key Learnings

- **`val bindings` must be per-eval, not per-session.** Declaring `val bindings = getBindings(ENGINE_SCOPE)` in
  only the first synthetic snippet stores a static field capturing that eval's ScriptContext. Subsequent
  evals with a different ScriptContext (e.g. `engine.eval(script, customBindings)`) see stale bindings.
  Fix: declare `val bindings` in every synthetic snippet so property getters reference their own class's
  `bindings` field, which is set from the ScriptContext active at that eval's `$$eval` invocation.
- **`getCurrentState(getContext())` not `getCurrentState(context)` for lineCounter.** Using `context`'s
  state when `context` differs from `getContext()` (e.g. custom Bindings passed to `eval(String, Bindings)`)
  creates a fresh state with lineCounter=0, colliding with existing snippets in the outer compiler's
  history. The outer default-context state is the correct lineCounter source since it co-ordinates with
  `replCompiler = getCurrentState(getContext()).compiler`.
- **`@InlineOnly` in generated snippet code.** Any use of `?.let{}`, `?.also{}`, `?.takeIf{}`, `?.apply{}`,
  `[] = value` (MutableMap.set) inside synthetic snippet Kotlin source fails at K2 REPL codegen with
  "Unhandled intrinsic FUN IR_EXTERNAL_DECLARATION_STUB [inline]". Must use explicit null checks and
  `.put(...)`.
- **JVM identifier constraint blocks `\`-based escaping.** `FirJvmNamesChecker.INVALID_CHARS` includes `\`.
  The `testEvalWithContextNamesWithSymbols` escape table (`.` → `\,`, `:` → `\!`, etc.) uses backslash
  in all escape sequences, so NO binding name with a JVM-forbidden char (`.`, `:`, `;`, `[`, `]`, `/`,
  `<`, `>`, `\`) can be exposed as a property. Needs a JVM-safe encoding scheme — out of scope for this
  iteration.
- **Pre-existing K2 REPL `ReplState.put/get [fake_override]` blocks custom-context binding tests.**
  `testEvalWithContext` and `testEvalWithContextDirect` are unblocked logically by the `lastScriptContext`
  fix, but the user snippet's `$$eval` triggers `ReplState.put [fake_override]` → codegen failure. This
  is the same codegen bug as the 4 pre-existing failures. The bug appears to be triggered when the user
  snippet accesses members across FIR module boundaries (e.g., `bindings["key"] as Int` where cast
  generates a FOLDED_ELVIS IR that references `kotlin.let`).
