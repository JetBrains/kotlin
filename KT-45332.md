# KT-45332

## Summary
MPP: "expect function has no actual declaration for JVM" with Android target — spurious `NO_ACTUAL_FOR_EXPECT` diagnostic raised on `expect` declarations even when `actual` declarations exist in `androidMain`/`androidTest` source sets.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt45332.kt`

## Observations
The issue described in KT-45332 is a Gradle build configuration / source set resolution problem: in a KMM project targeting Android and iOS, the compiler (or IDE analysis) incorrectly reports "Expected function has no actual declaration for JVM" even when `actual` declarations exist in the platform-specific source sets.

The FIR analysis test framework operates on single-file compilations without a full multiplatform Gradle project context, so the exact scenario (separate `commonMain`/`androidMain` source sets with Android target acting as JVM) cannot be reproduced in isolation.

In the created test, when `expect` and `actual` are placed in the same non-multiplatform compilation unit, the FIR compiler produces:
- `NOT_A_MULTIPLATFORM_COMPILATION` on the `expect` and `actual` keywords
- `EXPECT_AND_ACTUAL_IN_THE_SAME_MODULE` on the function names

These are different diagnostics from the ticket's reported `NO_ACTUAL_FOR_EXPECT`. The test does **not** reproduce the original bug (spurious "no actual for JVM" error in a properly configured MPP project), indicating the behavior in the FIR front-end itself is consistent — the root cause lies in how Kotlin Gradle Plugin configures module dependencies for Android test source sets, not in the FIR analysis logic directly.

Both test runs (with and without `--update-test-data`) exited with code 0.
