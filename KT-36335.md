# KT-36335

## Summary
AssertionError: Recursion detected in a lazy value under LockBasedStorageManager when compiling Kotlin code that calls methods on classes from the BouncyCastle library.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt36335.kt`

## Observations

The original issue is a K1 frontend crash (`AssertionError: Recursion detected in a lazy value under LockBasedStorageManager`) that occurs specifically when:
1. The code uses classes from binary dependencies (BouncyCastle JARs)
2. Those classes have a complex inheritance hierarchy with type parameters
3. The K1 `DeserializedMemberScope.OptimizedImplementation` triggers recursion while computing `functionNames` via `SubstitutingScope`

The FIR test was created with a representative code pattern that simulates similar inheritance hierarchies with generics (`AbstractCipher<T>` → `BlockCipher<T>` → `AESEngine`, etc.) and method calls through star-projected generic types (`BlockCipher<*>`).

The FIR test **passes cleanly** without any diagnostics or errors. This is expected because:
- The original bug is in the K1 frontend's `LockBasedStorageManager`/`DeserializedMemberScope` machinery, which FIR (K2) does not use
- FIR has a completely different architecture for resolving member scopes that avoids this kind of recursive lazy computation
- The test pattern with source-only classes cannot reproduce the recursion that was triggered by loading compiled class files from BouncyCastle JARs

Since FIR/K2 does not exhibit the bug (the code compiles cleanly in K2), this is `repro: false` — the bug is not present in K2.
