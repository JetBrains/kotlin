# KT-29877

## Summary
Null literals are trying to cast to not-null Nothing — wrong diagnostics when calling methods on `null` inside `if (null != null)` or `if (null is Int)` blocks.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt29877.kt`

## Observations
The original K1 issue reported "Smart cast to 'Nothing' is impossible, because 'null' is a complex expression" (SMARTCAST_IMPOSSIBLE) for `null.inv()` inside blocks conditioned on `null != null` or `null is Int`.

In FIR/K2, the diagnostics are:
- `SENSELESS_COMPARISON` for `null != null` (correct new diagnostic)
- `UNSAFE_CALL` on the receiver dot for `null.inv()` (this is the expected nullable receiver error)
- `MISSING_DEPENDENCY_CLASS` on `inv` (unexpected — this is the FIR manifestation of the bug)

The `MISSING_DEPENDENCY_CLASS` on `inv` when called on `null` inside a type-narrowing block is the FIR-equivalent of the original issue: the null literal is still being treated specially (as if castable to not-null `Nothing`) when type-narrowing is in effect, causing incorrect resolution of `inv()` that results in a bogus dependency class error rather than simply a nullable receiver error.
