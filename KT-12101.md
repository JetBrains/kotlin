# KT-12101

## Summary
Confusing error message for TYPE_PARAMETER_AS_REIFIED in 'arrayOf<T>' and similar calls

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt12101.kt`

## Observations
The FIR (K2) compiler still produces `TYPE_PARAMETER_AS_REIFIED` diagnostics for all three
constructs described in the ticket when a non-reified type parameter is used as a type argument
to `arrayOf`, `Array`, and `arrayOfNulls`:

- `arrayOf<E?>()` — reports `TYPE_PARAMETER_AS_REIFIED` on `E?`
- `Array<E?>(size = 1, init = { i -> null })` — reports `TYPE_PARAMETER_AS_REIFIED` on `E?`
- `arrayOfNulls<E>(6)` — reports `TYPE_PARAMETER_AS_REIFIED` on `E`

The first test run with `--update-test-data` automatically inserted the inline diagnostic markers
into the `.kt` file confirming this behavior. The verification run passed cleanly, confirming
the diagnostics match the current compiler output.

The issue (confusing error message referencing "reified" for stdlib functions that internally
use reified type parameters) is thus still observable in K2 FIR, even though the ticket is
currently marked "Can't Reproduce" based on playground/IDE testing.
