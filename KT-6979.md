# KT-6979

## Summary
SAM constructor should be treated as a constructor (not a function) by the compiler, to prevent ambiguous resolution when multiple classes with the same FQ name exist in the classpath.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt6979.kt`

## Observations
The issue described behavior where SAM constructors, being treated as functions rather than constructors,
could lead to ambiguous resolution: since multiple functions with the same name can be imported from
different packages, importing a SAM constructor could unintentionally construct a different class that
happened to appear earlier in the classpath.

The test was created with two `fun interface` types (`Foo` and `Bar`) and their SAM constructor calls.
In K2/FIR, the test compiles cleanly without any errors or warnings — SAM constructors resolve correctly
as constructors and no ambiguity occurs.

The ticket was marked as **Obsolete** with the comment "I think that makes no much sense in K2."
This confirms the issue is not present in K2 — the behavior described in the ticket (problematic
treatment of SAM constructors as functions leading to potential classpath ambiguity) does not manifest
in the FIR frontend. The bug is gone in K2.
