# KT-22520

## Summary
Expect functions are second-class to non-expect functions in overload resolution: in common multiplatform code, `foo(1)` resolves to the generic `fun <T> foo(x: T)` instead of the more specific `expect fun foo(x: Int)`, and `::foo` produces `OVERLOAD_RESOLUTION_AMBIGUITY`.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt22520.kt`

## Observations
The test was created as a single-file FIR analysis test. However, the reproducer code from the ticket requires a multiplatform compilation context where `expect` declarations are valid. In a non-multiplatform single-file test, FIR correctly rejects the `expect` keyword with `NOT_A_MULTIPLATFORM_COMPILATION` diagnostic before overload resolution can exhibit the buggy behavior.

The test runs and passes with `// RUN_PIPELINE_TILL: FRONTEND` and a `<!NOT_A_MULTIPLATFORM_COMPILATION!>expect<!>` diagnostic marker, but this does not reproduce the core bug (incorrect overload resolution preference for expect functions in multiplatform common code).

To properly reproduce KT-22520, a multiplatform module test with a `common` module would be needed, which is outside the scope of the `resolve/problems/` single-file test format.
