# KT-47428

## Summary
Unexpected "deimport" of nested class type parameter upper bound with star projections involved.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt47428.kt`

## Observations
The issue describes a situation where using a parametrized nested class (A) as a type argument for
an outer class (B) with star projections causes the type parameter bounds of that nested class to
become "deimported" in the IDE, triggering an import loop.

The underlying compiler-level problem (reported in the related ticket KT-38288) was an
`UNRESOLVED_REFERENCE` for the type parameter upper bound (e.g., `Any`) of a nested class when
the outer class extends another class using a star-projected type argument referencing that nested
class. Minimal reproducer:

```kotlin
open class A<K : Any>

class G : A<G.Key<*>>() {
    class Key<T : Any>  // K1: Error: Unresolved reference: Any
}
```

In the FIR/K2 frontend this code compiles successfully — no `UNRESOLVED_REFERENCE` is produced.
The test runs with `RUN_PIPELINE_TILL: BACKEND` and passes cleanly. This matches the
`not-reproducible-in-k2` tag on KT-38288 (fixed in 1.9.0 / K2).

The remaining open aspect of KT-47428 (IDE import cycling) is a tooling-level concern that
cannot be captured in a FIR analysis test.
