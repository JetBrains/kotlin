# KT-53267

## Summary
Nested typealias call with an upperbound doesn't typecheck correctly — `SimpleApi<Unit>()` fails to compile even though it is equivalent to `Api<Unit>()` which works.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt53267.kt`

## Observations
The FIR (K2) compiler successfully compiles both `val api = SimpleApi<Unit>()` and `val api2 = Api<Unit>()` without any diagnostic errors. The test runs with `RUN_PIPELINE_TILL: BACKEND` and passes cleanly.

This indicates the bug described in KT-53267 (incorrect rejection of nested typealias calls with type parameter upper bounds) is **not present in FIR/K2**. The issue was a K1 (legacy frontend) bug. In FIR, the type-checking of nested typealiases with upper bounds works correctly.
