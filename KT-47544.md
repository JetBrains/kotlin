# KT-47544

## Summary
`@kotlin.Deprecated(message = "", level = kotlin.DeprecationLevel.HIDDEN)` on Scala methods does not hide them from Kotlin overload resolution, causing "Overload resolution ambiguity" when calling from Kotlin.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt47544.kt`

## Observations
The test uses Java source files to simulate the scenario: a class with two overloaded methods where one is annotated with `@kotlin.Deprecated(message = "", level = kotlin.DeprecationLevel.HIDDEN)`. The Kotlin call `c.javaFunction { a, b -> a + b }` resolves cleanly without `OVERLOAD_RESOLUTION_AMBIGUITY`.

The test passes with `RUN_PIPELINE_TILL: BACKEND` and no error diagnostics, confirming that **K2/FIR correctly handles `@kotlin.Deprecated(level = HIDDEN)` for Java methods** — the hidden overload is excluded from resolution as expected.

The original bug is **Scala-specific**: when the same annotation is applied to a Scala method, it does not get hidden from Kotlin resolution. This is likely because Scala encodes annotations differently in JVM bytecode (or stores them only in the Scala signature rather than as a regular Java bytecode annotation), so the Kotlin compiler does not see the `HIDDEN` deprecation.

The Scala-specific case cannot be reproduced in a text-based FIR analysis test, as this test format only supports Java source files — not pre-compiled Scala bytecode. A proper reproduction would require including compiled `.class` files from Scala.

Since the Java case (which the test covers) shows correct behavior with no ambiguity, the bug as described in the ticket is **not reproduced** by this test.
