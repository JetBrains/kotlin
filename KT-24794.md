# KT-24794

## Summary
Problems resolving Groovy `@Delegate` methods in super class — Kotlin fails to call delegated methods when a Groovy class uses `@CompileStatic @Delegate`, producing "cannot be invoked as a function" and "Unresolved reference" errors.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt24794.kt`

## Observations
The issue was reported for K1 where calling methods on a Kotlin subclass of a Groovy `@CompileStatic @Delegate` class produced:
- `Expression 'delegated' of type 'DelegatedGroovy!' cannot be invoked as a function. The function 'invoke()' is not found`
- `Unresolved reference: delegatedParams`

The core problem is that Groovy's `@Delegate` generates methods (e.g., `delegated()`, `delegatedParams(String)`) whose names conflict with the property accessor pattern (`getDelegated()` maps to property `delegated` in Kotlin). K1 incorrectly resolved `delegating.delegated()` as property invocation instead of a method call.

A Java simulation was created replicating the same bytecode structure: `DelegatingGroovy` has both `getDelegated()` (property getter) and `delegated()` (delegate method) as separate public methods. The FIR/K2 compiler resolves all calls correctly — member functions take precedence over property invocation — and compiles the test to BACKEND with no errors. The generated tags confirm Java interop works: `javaFunction`, `javaType`.

The bug is not reproducible with K2/FIR: the compiler correctly prefers the member function `delegated()` over treating `delegated` as a property to be invoked. The issue appears to have been fixed in the K2 frontend.
