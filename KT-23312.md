# KT-23312

## Summary
Type inference with regard to expected type and 'Exact' constraints — `T` is inferred as `String?` instead of `String` when calling a function with `@Exact` return type annotation in a `String?` expected-type context.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt23312.kt`

## Observations
The test reproduces the scenario described in the ticket: when `elvis("non-null", "non-null")` is called in a `String?` expected-type context (assignment to a `String?` variable or `val` declaration), the `@Exact` annotation on the return type causes an additional constraint `T <: String?`, leading to `T = String?` even though both arguments are `String`.

The test compiles successfully with `RUN_PIPELINE_TILL: BACKEND` (no errors), confirming that the `String?` inference is accepted. The framework added an `ERROR_SUPPRESSION` diagnostic on the `"INVISIBLE_REFERENCE"` suppress annotation string (as expected for internal annotations), and generated the `GENERATED_FIR_TAGS` block. Both update and verification runs passed cleanly.

The bug is present: the code compiles without errors, which confirms the compiler accepts `T = String?` as the inference result, matching the buggy behavior described in the ticket.
