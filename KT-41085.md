# KT-41085

## Summary
MPP: Performance problem in `ExpectActualDeclarationChecker.checkActualDeclarationHasExpected` — the check is quadratic because every declaration is checked against all other declarations in the module.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt41085.kt`

## Observations
The test creates a multi-platform module scenario with:
- A `common` module containing an `expect fun testFun(): String` declaration
- A `main(common)` platform module containing the `actual` implementation and many private overloaded functions with the same name (`foo`)

The many private `foo` overloads represent the scenario described in the ticket (600 auto-generated declarations with the same name in the original report).

When running the test under K2/FIR, the compiler produces `ACTUAL_WITHOUT_EXPECT` for the `actual fun testFun()` declaration — FIR does not recognize the expect/actual pairing across the test's multi-module setup. This indicates the expect/actual matching in FIR for this multi-platform module configuration is not functioning correctly (the actual declaration has a corresponding expect in the common module, but FIR reports it as unmatched).

The original K1 performance issue (quadratic `checkActualDeclarationHasExpected`) is replaced in K2/FIR by a correctness issue: the expect/actual relationship is not established at all for this scenario. The test passes using `RUN_PIPELINE_TILL: FRONTEND` with the `ACTUAL_WITHOUT_EXPECT` diagnostic marker, confirming the buggy behavior is still observable in FIR.
