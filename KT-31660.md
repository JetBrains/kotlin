# KT-31660

## Summary
Confusing error message when the last statement in lambda is a declaration, not an expression — the entire `val x: Type = ...` declaration is highlighted with "Expected a value of type X", making it hard to understand the real problem.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt31660.kt`

## Observations
The test reproduces the described behavior in FIR/K2. When the lambda `{ val x: String = getSomeString() }` is assigned to a `() -> String` variable, FIR reports `RETURN_TYPE_MISMATCH` on the entire `val x: String = getSomeString()` declaration:

```
val implicitReturn: () -> String = {
    <!RETURN_TYPE_MISMATCH!>val x: String = getSomeString()<!>
}
```

This matches the ticket description: the whole `val x` declaration is highlighted, which is confusing because it makes users think the declaration itself is wrong, when the actual issue is that the lambda body does not end with an expression (the last statement is a declaration, so the lambda implicitly returns `Unit` instead of `String`). A clearer error would point to the lambda's closing brace or the missing return expression.

The test passes by asserting the current (broken) behavior — the bug is still present.
