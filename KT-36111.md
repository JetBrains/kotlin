# KT-36111

## Summary
False negative warning on exhaustive `when` with flexible type of sealed class in subject.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt36111.kt`

## Observations
In K1, when an exhaustive `when` expression uses a Java-returned value as its subject:
- **Enum from Java** (`E..E?` flexible type): the compiler correctly issues `WHEN_ENUM_CAN_BE_NULL_IN_JAVA` warning because the subject can be null but no null branch exists.
- **Sealed class from Java** (`A..A?` flexible type): no warning is issued — false negative (the bug described in KT-36111).

In K2/FIR, the situation is **worse**:
- **Enum from Java** (`E..E?`): no `WHEN_ENUM_CAN_BE_NULL_IN_JAVA` warning either — K2 does not implement this diagnostic at all.
- **Sealed class from Java** (`A..A?`): no warning — false negative confirmed in K2 too.

The test passes with no inline diagnostic markers for either `test()` or `test_2()`, showing that the FIR frontend produces no null-safety warnings for either case. The false negative described in the ticket (missing warning for sealed class) is present in K2/FIR.
