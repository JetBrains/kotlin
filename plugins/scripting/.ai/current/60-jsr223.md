## What works (K2)

- Compile and evaluate JSR-223 scripts as REPL snippets
- Cross-snippet references (history via `FirReplHistoryScope`)
- Invocable interface (call top-level functions in evaluated snippet)

## Parsing path

`K2ReplCompiler` uses **hybrid** parsing (line anchors in [`10-compiler-representation.md`](10-compiler-representation.md)):

- `KtFileScriptSource` inputs → `session.buildFirFromKtFiles` (PSI). Snippet `KtFile` obtained via `getScriptKtFile` (lines 259-264), marked with `markAsReplSnippet()`.
- Other `SourceCode` → `session.buildFirViaLightTree`.

Tracking issue: **KT-83498** — "implement LT support, similarly as for the scripting". Until landed, JSR-223 embedders that pass `KtFileScriptSource` (the typical case) keep PSI on the path.

## Known gaps (K2)

### 1. Bindings

JSR-223 `Bindings` map should expose its entries as variables inside the script. K1 path did this by injecting provided properties into `FirScript`/`LazyScriptDescriptor`. K2 REPL uses `FirReplSnippet` (different shape — embedded class + eval fn), so the binding-to-property injection needs porting through the snippet builder path.

Files involved (gap to close):
- `FirReplSnippetConfiguratorExtensionImpl` — needs to inject binding variables into the snippet class as ctor params or into the eval fn signature
- `KotlinJsr223ScriptEngineImpl` — needs to thread bindings through to the configurator

### 2. Remote compilation

K1 had a remote (daemon-based) compilation scenario for JSR-223 — was useful for sandboxing / process isolation. With daemon REPL slated for removal, the K2 remote scenario, if needed, would be Build Tools API based.

No code path exists today. **Open question**: keep this feature at all?

### 3. Classloader-reflection dependency extraction

Test `ResolveDependenciesTest` marked `expectTestToFailOnK2`. The K1 path could pull deps from a parent classloader by reflection; K2 doesn't yet.
