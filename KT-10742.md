# KT-10742

## Summary
Operator convention satisfied with a variable of function with receiver type, exception in back-end (subtask KT-18295 of meta-issue KT-10742: "Call conventions and operator overloading").

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt10742.kt`

## Observations
The reproducer is based on KT-18295, the specific open subtask cited in KT-10742's description.

The code uses a function-with-receiver parameter `iterator: T.() -> Iterator<*>` to (incorrectly) satisfy the `iterator` convention for a `for` loop over `args: T`.

In FIR (K2), the compiler correctly reports `NOT_FUNCTION_AS_OPERATOR` on `args` in the `for` loop, since a functional type cannot have the `operator` modifier and therefore cannot satisfy operator conventions.

The test passes with `// RUN_PIPELINE_TILL: FRONTEND` and the inline marker:
```
for (item in <!NOT_FUNCTION_AS_OPERATOR!>args<!>)
```

This confirms the bug is reproduced: the original K1 compiler accepted this code without errors (allowing it to reach the backend where an exception occurred), while FIR now correctly reports an error at the frontend.
