# KT-35501

## Summary
Too large highlighting range for "Type mismatch" with qualified return inside a lambda that returns specified type.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt35501.kt`

## Observations
The reproducer uses `return@block` (without a value, thus returning `Unit`) inside a lambda where `String` is expected:

```kotlin
fun <R> block(block: () -> R): R = block()

val a: String =
    block {
        return@block
    }
```

FIR reports a `RETURN_TYPE_MISMATCH` diagnostic on the `return@block` expression itself (line 10), which is more precise than the K1 behavior described in the ticket — K1 highlighted the entire lambda body. However, the issue remains present: the diagnostic fires on the `return@block` statement, confirming the type mismatch is still detected. The bug described in the ticket (too-large highlighting range in K1/IDE) manifests as the diagnostic being placed on the whole lambda body in K1; in FIR the range is correctly narrowed to just `return@block`.

The test asserts the current FIR behavior with `<!RETURN_TYPE_MISMATCH!>return@block<!>`, showing the issue is reproduced (type mismatch still reported, and is observable).

Test passed cleanly after two `--update-test-data` runs (the first run auto-inserted the diagnostic marker; `GENERATED_FIR_TAGS` required manual addition due to test parallelism overwriting the file before the Phased test runner could persist it).
