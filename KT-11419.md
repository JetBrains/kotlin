# KT-11419

## Summary
Additional call checkers — meta-issue tracking the implementation of various
additional checkers for call resolution, including proper `NON_TAIL_RECURSIVE_CALL`
reporting for `tailrec` functions called in non-tail positions.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt11419.kt`

## Observations
The test is based on sub-issue KT-27437 ("Tail-recursive call is not detected
in a nested lambda"), which reported that a recursive call to a `tailrec` function
inside a nested lambda was not being flagged as a non-tail call.

The scenario: in a continuation-passing-style height computation, the inner call
`heightCPS(t.right, fun(hr: Int): Int { ... })` is placed inside an anonymous
function, which means it is NOT in tail position of the outer `tailrec` function.

K2/FIR correctly identifies this and reports `NON_TAIL_RECURSIVE_CALL` on the
inner call:

```kotlin
return heightCPS(t.left, fun(hl: Int): Int {
    return <!NON_TAIL_RECURSIVE_CALL!>heightCPS<!>(t.right, fun(hr: Int): Int {
        return k(if (hl > hr) hl else hr)
    })
})
```

The outer call to `heightCPS` (the one that passes the first anonymous function
as `k`) remains in tail position and is correctly NOT flagged.

The test passes in all relevant test runners:
- `FirLightTreeDiagnosticsWithLatestLanguageVersionTestGenerated` — SUCCESS
- `PhasedJvmDiagnosticLightTreeTestGenerated` — SUCCESS
- `PhasedJvmDiagnosticPsiTestGenerated` — SUCCESS
- `FirLightTreeDiagnosticsWithoutAliasExpansionTestGenerated` — SKIPPED (no typealias)
- All `LLDiagnosticsFirTestGenerated` variants — SUCCESS

K2 correctly implements the `NON_TAIL_RECURSIVE_CALL` checker for calls inside
nested lambdas. The bug (missing diagnostic in K1) is not reproduced in K2.
The meta-issue KT-11419 remains open as it tracks multiple sub-items, but the
`tailrec`-in-nested-lambda checker works correctly in K2.
