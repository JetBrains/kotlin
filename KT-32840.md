# KT-32840

## Summary
Declarations clash in hierarchical multiplatform projects on override: when `actual interface Input` defines `fun foo()`, and `abstract class AbstractInput : Input` (common code) also defines `fun foo()`, the FIR compiler reports `VIRTUAL_MEMBER_HIDDEN` diagnostic for the class method.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt32840.kt`
`compiler/fir/analysis-tests/testData/resolve/problems/kt32840.ll.kt` (LL FIR divergence file)

## Observations
The FIR K2 compiler detects the clash and reports `VIRTUAL_MEMBER_HIDDEN: 'foo' hides member of supertype 'Input' and needs an 'override' modifier` for `AbstractInput.foo()` in the common module. This occurs because the platform-specific `actual interface Input` provides a default implementation of `fun foo(): Int = 2`, which clashes with `AbstractInput`'s own `fun foo(): Int = 1`.

Key findings:
- The `VIRTUAL_MEMBER_HIDDEN` diagnostic (an error) is produced by the FIR compiler for `AbstractInput.foo()` in the common module.
- The LL FIR API does **not** produce this diagnostic (LL_FIR_DIVERGENCE), hence the separate `kt32840.ll.kt` file.
- The `PhasedJvmDiagnosticLightTreeTestGenerated` and `PhasedJvmDiagnosticPsiTestGenerated` tests pass once `// FILE:` directives are removed from the module declarations.
- The test uses `// RUN_PIPELINE_TILL: FRONTEND` since the `VIRTUAL_MEMBER_HIDDEN` error prevents backend compilation.

The bug described in KT-32840 is reproduced: the K2 FIR compiler correctly detects the declaration clash that the issue requested checking for.
