# KT-16467

## Summary
No compiler error for inline function lambda parameter called from anonymous object on implicit receiver; K1 compiler crashed with IndexOutOfBoundsException.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt16467.kt`

## Observations
The reproducer code uses an inline lambda parameter `f` passed as a receiver to `apply {}`, with an anonymous object inside that `init` block calls `invoke()` via the implicit receiver (which is `f`).

In K1, this produced no compiler error and crashed with `java.lang.IndexOutOfBoundsException: Cannot pop operand off an empty stack` in the bytecode analysis stage.

In K2/FIR, the compiler correctly reports a `USAGE_IS_NOT_INLINABLE` diagnostic on `f` in `f.apply {}`:
```
USAGE_IS_NOT_INLINABLE: Illegal usage of inline parameter 'f: () -> Unit'. Add 'noinline' modifier to the parameter declaration.
```

K2 catches the issue at the `f.apply {}` boundary (using an inline lambda in a non-inlinable context), which subsumes the inner problem. The code is rejected with a proper error and no crash occurs. The bug (no error + compiler crash) is not reproducible in K2.

Note: K2 uses `USAGE_IS_NOT_INLINABLE` rather than `NON_LOCAL_RETURN_NOT_ALLOWED` as was suggested in the ticket, but either way the illegal usage is correctly diagnosed and the code is rejected.
