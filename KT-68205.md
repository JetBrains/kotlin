# KT-68205

## Summary
Consider other approaches for handling ILT (Integer Literal Type) types in ResultTypeResolver — questions whether K2's special-casing of ILT in equality constraints in `findResultTypeOrNull` is correct.

## Test file
`compiler/fir/analysis-tests/testData/resolve/problems/kt68205.kt`

## Observations
The test exercises various ILT constraint resolution scenarios in K2:
- Simple ILT inference (e.g., `val a = id(1)` → infers `Int`)
- ILT with a directional constraint from expected type (e.g., `val b: Long = id(1)` → infers `Long`)
- Equality constraints from multiple ILT literals (e.g., `same(1, 2)` → infers `Int`)
- ILT equality + directional constraint (e.g., `val d: Long = same(1, 2)` → infers `Long`)
- Mixed ILT and concrete Long (e.g., `same(1, 2L)` → infers `Long`)
- Nested ILT (e.g., `id(id(1))` → infers `Int`)
- ILT with Byte and Short directional constraints

All cases compile successfully to BACKEND with no errors or unexpected diagnostics. The K2 code in `ResultTypeResolver.findResultTypeOrNull` that treats ILT-containing equality constraints differently (by not returning immediately and instead considering the directional type) produces correct results for these standard scenarios. The potential correctness concern raised in the ticket — that choosing `resultTypeFromDirection` when it is a proper subtype might violate constraints — is not observable as wrong behavior in practice.

The issue is a code quality / cleanup task (tagged `k2-cleanup`) questioning the correctness and rationale of the ILT special-casing logic. Since the fix described in the ticket (clarifying or reworking the logic) does not yet appear to have been implemented, but the observable behavior is correct, `repro: false` — the code paths in question work as expected with no wrong type inference visible in test output.
